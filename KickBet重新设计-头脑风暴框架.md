## 核心定位

**从"工具"转型为"教育型助手"**

> KickBet 不是告诉你投什么，而是教你理解为什么。

### 模块 2："为什么"解释系统

**分层解释设计：**

```
【小白层】一句话结论
  "曼城胜率70%，赔率1.42有价值，建议投注"

【中级层】三段式解释
  1. 数据依据：曼城近10场场均2.1球，布伦特福德客场防守弱
  2. 赔率分析：1.42赔率隐含概率70%，模型计算概率75%，有5%优势
  3. 博彩公司意图：Bet365开出1.42可能是吸引主队投注，需警惕

【高级层】完整分析报告
  - Poisson模拟5000次结果
  - Kelly资金比例计算
  - 赔率历史变动追踪
  - 市场情绪分析
```

**交互式问答方案：**

```
用户点击"为什么？"按钮 →

Q: 为什么推荐曼城？
  → 点击展开详细数据

Q: 为什么赔率1.42有价值？
  → 点击展开概率对比

Q: 博彩公司为什么调赔率？
  → 点击展开变动历史
```

**可视化图表方案：**

```
[概率对比图]
模型概率: 75%
赔率隐含: 70%
优势空间: 5%

[赔率变动图]
赛前3天: 1.35
赛前1天: 1.42 ↑ (+5%)
临场: 1.48 ↑ (+3%)
```

### 模块 3：赔率变动追踪

**博彩公司调赔率的常见原因：**

| 原因 | 说明 | 用户关心度 |
|------|------|------------|
| **资金流向** | 大量资金投主队 → 赔率下降 | ⭐⭐⭐⭐⭐ |
| **伤病消息** | 主力球员伤缺 → 赔率上升 | ⭐⭐⭐⭐⭐ |
| **市场情绪** | 公众看好某队 → 赔率调整 | ⭐⭐⭐⭐ |
| **平衡账面** | 博彩公司规避风险 | ⭐⭐⭐ |
| **诱导投注** | 刻意设置诱饵赔率 | ⭐⭐⭐⭐⭐ |

**解释方式：**

```
【赔率变动追踪】
  开盘: 1.35 (赛前3天)
  当前: 1.42 (赛前1天)
  变动: +5%

【可能原因分析】
  ⚠️ 资金流向: 主队投注量增加35%
  ⚠️ 伤病影响: 客队主力后卫复出
  ⚠️ 市场情绪: 公众看好主队大胜

【博彩公司意图推测】
  1.42可能是在吸引主队投注，需警惕
  如果临场降到1.38，说明资金继续流入
  如果临场升到1.48，可能是诱导反向投注
```

## 四、技术架构设计

### 当前架构问题

| 问题 | 说明 |
|------|------|
| **数据源单一** | 只依赖 Odds-API.io，Rate Limit 限制 |
| **无持久化** | 预测历史不保存 |
| **无用户系统** | 无法区分免费/付费用户 |
| **无前端** | 只有 API，没有界面 |

---

### 新架构建议

```
数据层
├── 多数据源（Odds-API.io + football-data + 自建爬虫）
├── 数据缓存（Redis）
├── 历史存储（SQLite/PostgreSQL）
└── Rate Limit 监控

业务层
├── 用户系统（认证 + 权限）
├── 预测引擎（Poisson + Kelly）
├── 分析引擎（赔率变动追踪）
├── 内容系统（教育内容 + 案例库）
└── 追踪系统（用户投注记录）

展示层
├── Web前端（移动端优先）
├── API接口（第三方接入）
└── 管理后台（内容管理）
```

---

### 数据源多样性方案

| 数据源 | 用途 | 优势 | 成本 |
|--------|------|------|------|
| **Odds-API.io** | 欧洲5大联赛 + 中超 | Sbobet让球盘 | 免费100/h |
| **Football-Data.org** | 积分榜、球队统计 | 官方数据 | 免费 |
