## 一、数据分层缓存策略

### 1.1 三层缓存架构

```
┌─────────────────────────────────────────────────────────────┐
│ 基础数据层 (比赛、队伍信息)                                    │
│ - 更新周期: 1小时                                             │
│ - 数据特点: 相对稳定，比赛确定后基本不变                        │
│ - 数据源: Football-Data.org                                  │
│ - API限制: 10次/分钟                                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 蔬率数据层 (各平台蔬率)                                        │
│ - 更新周期: 10分钟 (基础) / 动态调整                           │
│ - 数据特点: 动态变化，临近比赛变化更频繁                        │
│ - 数据源: Odds-API.io                                        │
│ - API配额: 5,000次/小时                                       │
│ - Bookmaker限制: 最多2个 (Bet365 + Sbobet)                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ 预测结果层 (Poisson模型输出)                                   │
│ - 生成时机: 比赛确定 + 蔬率开出后                              │
│ - 数据特点: 预测结果对所有用户相同                             │
│ - 存储: 数据库持久化                                          │
│ - 无需重复计算                                                │
│ - 版本管理: 支持追溯变化历史                                   │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计优点

**降低API调用压力**
- 缓存后前端只需读取本地数据库
- API调用次数与用户数量无关
- 5,000次/小时配额完全够用

**预测结果缓存**
- Poisson模型预测对同一场比赛结果固定
- 避免每次用户访问都跑5000次模拟
- 大幅降低服务器计算负载

## 三、蔬率变化分析策略

### 3.1 数据存储策略

```
每次获取蔬率都存入数据库（带时间戳）

odds_history 表:
├── match_id: 比赛ID
├── bookmaker: 博彩公司

├── away_odds: 客胜蔬率

└── change_detected: 是否检测到变化

分析模块读取历史记录:
├── 比较最新两条记录的蔬率差异
├── 变化幅度 >5% 标记为显著变动
└── 显著变化时触发重新预测或AI分析
```

### 3.2 变化检测逻辑

```python
def detect_odds_change(match_id):
    # 获取最近两条蔬率记录
    latest = get_latest_odds(match_id)
    previous = get_previous_odds(match_id, latest.recorded_at)
    
    changes = {}
    for bet_type in ['home', 'draw', 'away']:
        old_odds = previous[f'{bet_type}_odds']
        new_odds = latest[f'{bet_type}_odds']
        change_pct = abs(new_odds - old_odds) / old_odds * 100
        
        if change_pct > 5:  # 5%阈值
            changes[bet_type] = {
                'old': old_odds,
                'new': new_odds,
                'change_pct': change_pct
            }
    
    return changes
```

## 八、Kelly Criterion 前端计算策略

### 8.1 设计核心理念

```
用户设计确认:

问题1: 用户选择比赛数量、资金不同，是否需要实时重算？
答案: ✓ 是，每次用户操作都需要重新计算分配

问题2: 计算是否只需从数据库获取预测数据？
答案: ✓ 是，预测数据(概率、蔬率)从数据库获取

问题3: 前端能否完成计算？
答案: ✓ 是，Kelly公式简单，前端完全可以计算

设计原理:
├── 预测数据: 数据库提供，不需要重新计算
├── 用户输入: 用户个性化选择
├── 计算执行: 前端完成，无需后端参与
└── 数据流: 一次API请求获取预测数据，后续本地计算
```

### 8.2 数据分层

```
组合计算的两种数据:

┌─────────────────────────────────────────────────────────────┐

│                                                             │

│                                                             │
│ 这是简单数学运算，前端完全可以完成                             │
└─────────────────────────────────────────────────────────────┘
```

### 8.3 Kelly公式详解

```
Kelly Criterion 基础公式:

Kelly比例 = (预测概率 - 隐含概率) / (蔬率 - 1)

简化版:
Kelly比例 = (p - 1/odds) / (odds - 1)

其中:
p = 模型预测概率 (如: 平局概率 21.7%)
odds = 蔬率 (如: 平局蔬率 4.50)
隐含概率 = 1/odds = 1/4.50 = 22.2%

计算示例:
比赛: 曼城 vs 水晶宫
投注类型: 平局
蔬率: 4.50
预测概率: 21.7%

Kelly计算:
隐含概率 = 1/4.50 = 22.2%
边缘优势 = 21.7% - 22.2% = -0.5% (负值，无价值)

如果蔬率是 5.0:
隐含概率 = 1/5.0 = 20%
边缘优势 = 21.7% - 20% = 1.7%
Kelly比例 = 1.7% / (5.0 - 1) = 1.7% / 4 = 0.43%
```

### 8.4 组合Kelly分配算法

```javascript
// 前端组合计算核心函数
function calculateComboAllocation(matches, budget, riskPreference) {
  
  // 1. 风险系数
  const riskFactor = {
    'conservative': 0.25,  // Quarter-Kelly
    'balanced': 0.5,       // Half-Kelly (推荐)
    'aggressive': 1.0      // Full-Kelly (风险高)
  };
  
  // 2. 计算每场的Kelly比例
  const kellyValues = matches.map(m => {
    const prob = m.predicted_prob;  // 从数据库获取
    const odds = m.best_odds;       // 从数据库获取
    
    // Kelly公式
    const edge = prob - (1/odds);
    let kelly = edge / (odds - 1);
    
    // 风险偏好调整
    kelly = kelly * riskFactor[riskPreference];
    
    // 只投注有价值的(Kelly > 0)
    if (kelly > 0) {
      return { match: m, kelly: kelly };
    }
    return null;
  }).filter(k => k !== null);
  
  // 3. 计算总Kelly
  const totalKelly = kellyValues.reduce((sum, k) => sum + k.kelly, 0);
  
  // 4. 分配预算
  const allocations = kellyValues.map(k => {
    const amount = budget * k.kelly / totalKelly;
    const expectedReturn = amount * k.match.best_odds;
    
    return {
      match_id: k.match.id,
      bet_type: k.match.bet_type,
      odds: k.match.best_odds,
      kelly_ratio: k.kelly,
      amount: Math.round(amount, 1),
      expected_return: Math.round(expectedReturn, 1)
    };
  });
  
  return allocations;
}
```

### 8.6 Half-Kelly策略说明

```
实际应用中使用Half-Kelly:

Full-Kelly风险较高:
├── 波动太大，容易亏损
├── 需要精确的概率估计
└── 实际应用容易超出最优

Half-Kelly更稳定:
├── 长期收益更平滑
├── 大多数专业投注者使用Half-Kelly
└── 减少极端波动的影响

风险系数定义:
├── 保守模式: Kelly × 0.25 (Quarter-Kelly)
├── 平衡模式: Kelly × 0.5 (Half-Kelly, 推荐)
├── 激进模式: Kelly × 1.0 (Full-Kelly)
└── 自定义: 用户可输入0.1-2.0之间的系数
```
