# KickBet 产品设计讨论记录

## 2026-05-13 讨论

### 1. 数据缓存策略

**分层更新周期：**
- 基础数据（比赛、队伍）：1小时更新一次
- 蔬率数据：10分钟更新一次
- 预测结果：比赛确定+蔬率开出后生成，存入数据库

**预测结果缓存原则：**
- Poisson模型预测对同一场比赛结果固定
- 预测结果对所有用户相同
- 缓存后无需每次用户访问重新计算
- 降低服务器负载，提高响应速度

**优化建议：**
- 根据比赛时间动态调整更新频率（临场高频更新）
- 预测结果版本管理（赔率变化触发重新预测）

### 2. 组合计算策略

**核心问题：用户组合比赛资金分配**

**结论：前端完全可以完成计算**

**计算拆分：**

```
第一部分：预测数据获取（数据库）
├── 每场比赛概率
├── 最优蔬率
└── 基础Kelly比例

第二部分：用户个性化计算（前端）
├── 选择哪些比赛
├── 总预算金额
├── 风险偏好调整
└── 资金分配计算
```

**优势：**
- 一次API请求获取数据
- 用户调整预算即时响应
- 无需每次调整都请求后端
- 降低服务器负载

### 3. Kelly Criterion 资金分配系统设计

**用户设计想法确认：**

用户提出的设计思路：
> 用户选择比赛数量、输入资金不同，是否需要实时重新计算？
> 计算是否可以通过从数据库获取已知预测数据，只是做数学计算？
> 这个计算是否可以在前端完成？

**确认结论：全部正确**

| 问题 | 答案 | 说明 |
|------|------|------|
| 用户改变选择需重新计算？ | ✓ 是 | 每次增减比赛、改预算、改风险偏好都需重算 |
| 计算只需数据库预测数据？ | ✓ 是 | 数据库提供概率+赔率，剩下纯数学运算 |
| 前端能否完成计算？ | ✓ 是 | Kelly公式是简单乘除，JavaScript完全胜任 |

**设计原理确认：**

用户对Kelly算法的理解：
> Kelly就是根据预测概率和赔率，计算每场比赛应该投注多少比例的资金

**理解正确**。Kelly Criterion核心逻辑：
1. 计算每场比赛的"价值"（模型概率 vs 赔率隐含概率）
2. 根据价值大小决定投注比例
3. 价值越高分配越多，价值为负不投

---

**公式：**
```
Kelly比例 = (预测概率 - 隐含概率) / (蔬率 - 1)
隐含概率 = 1 / 蔬率

示例：
预测概率 = 21.7%
蔬率 = 5.0
隐含概率 = 20%
Kelly = (21.7% - 20%) / (5.0 - 1) = 0.43%
```

**风险偏好调整：**
- 保守模式：Kelly × 0.25
- 平衡模式：Kelly × 0.5 (Half-Kelly)
- 激进模式：Kelly × 1.0 (Full-Kelly)

**组合分配逻辑：**
```
1. 计算每场Kelly比例
2. 计算总Kelly
3. 按比例分配预算
   单场金额 = 总预算 × 该场Kelly / 总Kelly
4. 计算预期回报
```

**串关建议：**
- 双串：预算的10-15%
- 三串：预算的5-10%
- 多串：预算的3-5%
- 不建议Full-Kelly做串关

### 4. Odds-API.io 配额修正与优化策略

**重要发现：配额信息修正**

之前记录: 100次/小时 (双Key 200次/小时)
实际配额: **5,000次/小时** (免费计划)

这是一个巨大的差异，之前的信息是错误的。

**实际限制：**
- 最多2个bookmakers (当前已选: Bet365 + Sbobet)
- 一次API调用获取指定bookmakers的蔬率数据
- 可更换bookmakers，但最多同时2个

**配额消耗重新计算：**

```
覆盖5大联赛场景:
├── 英超: events 1次 + odds 10次 = 11次
├── 西甲: events 1次 + odds 10次 = 11次
├── 意甲: events 1次 + odds 10次 = 11次
├── 德甲: events 1次 + odds 9次 = 10次
├── 法甲: events 1次 + odds 10次 = 11次

一次完整更新: 54次API调用
配额: 5,000次/小时

10分钟周期 (6次/小时): 324次 → 配额充足！
```

### 5. 蔬率变化分析策略

**用户补充建议：**

> 每次获取蔬率都存入数据库（带时间戳）
> 分析模块读取历史记录，检测变化
> 显著变化时才触发AI新闻分析（节省LLM调用）

**采纳设计：**

```
蔬率存储策略:
├── 每次获取蔬率存入odds_history表
├── 带时间戳recorded_at
├── 变化检测: 比较最近两条记录
├── 阈值: >5%标记为显著变动
└── >10%触发重新预测

L4等级蔬率变化分析:
├── 仅premium用户提供
├── 显著变化时调用LLM分析
├── 分析变化原因和市场因素
└── 提供投注策略影响建议
```

### 6. API调用智能调度

**用户补充建议：**

> 根据比赛时间动态调整
> 远场比赛低频更新，近场比赛高频更新
> 在配额内优先保证重要时段

**采纳设计：**

```
动态更新周期:
├── >24小时: 30分钟 (2次/小时)
├── 12-24小时: 20分钟 (3次/小时)
├── 6-12小时: 10分钟 (6次/小时)
├── 1-6小时: 5分钟 (12次/小时)
└── <1小时: 3分钟 (20次/小时)

智能调度算法:
├── 计算每场比赛距离开始时间
├── 分配到对应优先级组
├── 配额分配优先保证临场比赛
```

### 7. 预测结果版本管理

**用户补充建议：**

> predictions表加版本管理
> 前端只取is_current=true的版本
> 历史保留用于追溯预测变化

**采纳设计：**

```
predictions表字段:
├── match_id: 比赛ID
├── prediction_version: 版本号(v1,v2...)
├── created_at: 创建时间
├── trigger_reason: 触发原因
├── prediction_data: JSON数据
└── is_current: 是否当前版本

trigger_reason取值:
├── 'first_generation': 首次生成
├── 'odds_change': 蔬率变化触发
├── 'news_impact': 新闻影响触发
├── 'pre_match_adjust': 临场调整
└── 'manual_refresh': 手动刷新
```
