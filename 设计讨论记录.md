# KickBet 产品设计讨论记录

## 2026-05-13 讨论

### 1. 数据缓存策略

**分层更新周期：**
- 基础数据（比赛、队伍）：1小时更新一次
- 蔬率数据：10分钟更新一次
- 预测结果：比赛确定+蔬率开出后生成，存入数据库

**预测结果缓存原则：**
- Poisson模型预测对同一场比赛结果固定
- 预测结果对所有用户相同
- 缓存后无需每次用户访问重新计算
- 降低服务器负载，提高响应速度

**优化建议：**
- 根据比赛时间动态调整更新频率（临场高频更新）
- 预测结果版本管理（赔率变化触发重新预测）

### 2. 组合计算策略

**核心问题：用户组合比赛资金分配**

**结论：前端完全可以完成计算**

**计算拆分：**

```
第一部分：预测数据获取（数据库）
├── 每场比赛概率
├── 最优蔬率
└── 基础Kelly比例

第二部分：用户个性化计算（前端）
├── 选择哪些比赛
├── 总预算金额
├── 风险偏好调整
└── 资金分配计算
```

**优势：**
- 一次API请求获取数据
- 用户调整预算即时响应
- 无需每次调整都请求后端
- 降低服务器负载

---

### 3. Kelly Criterion 资金分配

**公式：**
```
Kelly比例 = (预测概率 - 隐含概率) / (蔬率 - 1)
隐含概率 = 1 / 蔬率

示例：
预测概率 = 21.7%
蔬率 = 5.0
隐含概率 = 20%
Kelly = (21.7% - 20%) / (5.0 - 1) = 0.43%
```

**风险偏好调整：**
- 保守模式：Kelly × 0.25
- 平衡模式：Kelly × 0.5 (Half-Kelly)
- 激进模式：Kelly × 1.0 (Full-Kelly)

**组合分配逻辑：**
```
1. 计算每场Kelly比例
2. 计算总Kelly
3. 按比例分配预算
   单场金额 = 总预算 × 该场Kelly / 总Kelly
4. 计算预期回报
```

**串关建议：**
- 双串：预算的10-15%
- 三串：预算的5-10%
- 多串：预算的3-5%
- 不建议Full-Kelly做串关
